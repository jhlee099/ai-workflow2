청중 여러분께

따라오시면서 보실 수 있어요. 각 단계가 왜 이렇게 연결되는지, 어떤 판단이 들어가는지에 집중해주세요.

핵심 포인트

모든 단계에서 제가 직접 코드를 쓰는 순간이 없어요. 제가 하는 건 방향 잡기와 판단뿐입니다.

STEP 1 / 7

Figma 디자인 화면

시작점은 기존 Figma 디자인이에요.
디자이너라면 이미 갖고 있는 화면 — 여기서부터 시작합니다.
특별한 준비 필요 없어요.

보여드릴 것

실제 Figma 파일을 열고, 화면 구조를 간단히 설명합니다. 어떤 화면인지, 무엇을 구현하려는지 맥락을 잡는 단계예요.

포인트

이 화면이 곧 코드로 변환될 재료예요. 지금은 그냥 시각적 시안이지만, 다음 단계에서 살아있는 코드가 됩니다.

STEP 2 / 7
Figma 디자인 → Figma Make → Claude Code → 방향 설정 → POC → 고도화 → 정책서

Figma Make에서 코드 뽑기

Figma Make는 디자인을 코드로 변환해줘요.
완벽하지 않아도 괜찮아요.
베이스 코드만 있으면 Claude Code가 나머지를 합니다.

보여드릴 것

Figma Make에서 화면 선택 → 코드 생성 → 결과물 확인. 생성된 코드를 복사하거나 파일로 저장합니다.

왜 Figma Make인가

처음부터 코드를 짜는 것보다 빠르고, 디자인 의도가 구조에 반영돼요. Claude Code에 넘길 베이스라인을 확보하는 게 목적입니다.

STEP 3 / 7
Figma 디자인 → Figma Make → Claude Code 가져오기 → 방향 설정 → POC → 고도화 → 정책서

Claude Code로 가져오기

Figma Make에서 뽑은 코드를 Claude Code 프로젝트로 가져와요.
이미 AIP 로드맵, 정책, 기획 맥락이 들어있는 폴더예요.
Claude는 그 맥락 위에서 코드를 이해합니다.

보여드릴 것

코드 파일을 프로젝트 폴더에 추가하고, Claude Code를 실행해서 현재 코드 구조를 파악시키는 과정이에요.

여기서 맥락이 중요해요

CLAUDE.md와 memory에 이미 담긴 AIP 로드맵·정책이 Claude가 "이 코드를 어떤 방향으로 발전시킬지" 판단하는 근거가 됩니다.

이게 핵심이에요

빈 화면 앞에서 혼자 고민하는 게 아니에요. 이미 쌓인 맥락 위에서 Claude가 첫 방향을 잡아주고, 저는 그걸 검토하고 결정합니다.

STEP 5–6 / 7
Figma 디자인 → Figma Make → Claude Code → 방향 설정 → POC 설계 + 고도화 → 정책서

POC 만들고
바로 고도화

방향이 잡히면 바로 만들어요.
첫 번째 결과물 확인 → 피드백 → 다시 고도화.
이 사이클이 대화처럼 이어집니다.

POC 설계

기반 코드에 방향을 적용해서 동작하는 첫 번째 결과물을 만들어요. "이 정도면 됩니다"가 아니라 "이 방향이 맞는지" 확인이 목적입니다.

고도화

첫 POC를 보고 "이건 이렇게, 저건 저렇게" 대화하듯 요청하면 Claude가 반영해요. 처음부터 다시 만들 필요가 없어요.

STEP 7 / 7

화면 → 문서 → 공유가 한 흐름 안에 있습니다.

보여드릴 것

"지금 만든 화면 기반으로 정책서 작성하고 Confluence에 올려줘" — 자연어 한 줄이면 됩니다. 문서 탭 열고, 붙여넣고, 저장하는 과정이 없어요.

이 흐름의 핵심

시작부터 끝까지 컨텍스트가 끊기지 않아요. Figma 화면이 코드가 되고, 코드가 정책서가 되고, 정책서가 Confluence에 올라갑니다. AI가 중간을 모두 이어줬어요.

터미널 기반이에요

명령줄 인터페이스(CLI)예요.
설치부터 실행까지
터미널을 벗어날 수 없어요.

비개발자한테는 진입 장벽
npm install -g @anthropic-ai/claude-code
cd [내 프로젝트 폴더]
claude

Node.js 설치, 경로 이동, 명령어 입력… 처음 보면 막막해요.

익숙하지 않으면 멈추는 지점

에러 메시지, 환경 변수, 권한 문제… 개발자가 아니라면 첫 실행부터 막혀요. 이게 초반 허들의 실체예요.

사실 익숙해지면 터미널도 할만해요. 명령어 몇 개만 알면 오히려 더 빠를 때도 있어요. 처음엔 앱으로, 나중에 터미널로 넘어가도 충분해요.

프로젝트 개요, 주제별 참조 파일 경로, 라우팅 표. 이 세 가지만 있어도 Claude가 필요한 파일만 골라 읽어요. 불필요한 읽기 = 토큰 낭비.

시작 명령어

"Claude Code 잘 활용하는 방법 리서치해서 내 폴더 구조에 적용할 계획 세워" — 이 한 문장으로 시작하면 Claude가 구조를 제안해줘요.

4

기획보다
POC 먼저

POC 먼저 만들면
기획서 수정 지옥이 없어요

기획서를 먼저 쓰면 글만 보고 판단해요. POC를 먼저 만들면 사용성 검증이 자동으로 돼요.

왜 더 좋냐면

POC를 만들면서 "아, 이 흐름이 어색하네"를 실제 화면에서 발견하게 돼요. 기획서 글자만 읽을 때는 못 잡는 것들이 만들다 보면 보여요.

순서

POC 먼저 만들고 → 써보면서 흐름 검증 → 기획서는 역으로 작성. 화면에서 검증된 내용이니 기획서도 훨씬 탄탄해져요.

새 POC = 파일 복사 + 기능만 추가.

방법

Chat 화면, 리스트 화면 등 자주 쓰는 UI 패턴을 HTML 파일로 만들어둬요. 새 POC는 파일 복사 → 기능만 추가. 처음부터 다시 만드는 시간이 없어요.

주의사항

베이스 파일은 반드시 GitHub에 저장해두세요. 로컬만 있으면 Claude Code가 폴더 정리할 때 사라질 수 있어요.

팁 8 / 11
8

수정사항 묶어서
단계별 처리

"3단계 계획 세워서 처리해줘"
= 누락 없이 토큰도 절약

수정사항을 하나씩 요청하면 비효율적이고, 한꺼번에 다 던지면 누락이 생겨요.

하나씩 요청 (비효율)

요청을 하나씩 던지면, 코드를 그때마다 다시 읽어요. 토큰도 낭비고, 시간도 낭비예요.

정답: 묶어서 단계별 계획

10개 주고 "3단계 계획 세워서 처리해줘"
= 코드 1번 읽고 3/3/4 단계로 처리. 누락 없이 효율적이에요.

상태창에 넣을 것들

상태창이 필요한 이유

"이 상태에서는 이렇게 돼요"를 클릭 한 번으로 보여줄 수 있어요. 말로 설명하는 것보다 10배 빠르게 이해시켜요.
